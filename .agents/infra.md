# Infrastructure Rules

## Hosting & processes
- Runs **natively on macOS**. No Docker, no DDEV. The dashboard backend must reach the host-authenticated `claude` and `codex` CLIs and the local filesystem directly — a container boundary would break that.
- Keep long-running processes alive with **pm2** or **launchd**, not a container.
- The dashboard is a **local** app (localhost). Never expose it publicly — it can read all accounts and (eventually) place trades.
- **Scheduled routines:** one launchd job per routine runs `scripts/run-routine.sh <id>`, which POSTs to `/api/routines/<id>` on the always-on dashboard server (the server runtime owns the lockfile, the code-gated execution pipeline, Alpaca paper, and the `codex` red-team). The endpoint runs under a single-instance lockfile and writes a `RunLog` to `data/logs/`. The trigger endpoint places paper orders, so it's gated by `ROUTINE_TRIGGER_TOKEN` (bearer) and must stay localhost-only. `scripts/install-routines.sh` writes the plists but never loads them — loading is a deliberate human act. **Execution is code-gated: every order clears the risk rails + red-team before Alpaca; the LLM proposes, it never executes.** **The autonomous execution batch (`executePendingProposals`) is paper-only:** it filters out every `account: "live"` proposal (approvable or advisory) so a live-intent order is **never** auto-placed — live orders go solely through the per-trade human approval path (`submitTradeApproval`). **Live-book risk management (Phase 3 M1, `routines/live-position-management.md`):** the live analog of `midday-scan`. A read-only `claude -p` routine reviews the latest `account: "live"` snapshot's holdings against their thesis/stop/take-profit (from the journal) and writes `account: "live"`, `advisory: false` **sell** (exit/trim) proposals for the human to approve → place. It opens **no new ideas** (pre-market's job) and **places nothing**; exits are never rail-blocked (a sell reduces risk) and an approved exit still routes to the dry-run sink until both gates open. Scheduled Mon–Fri 12:35 ET (just after the paper midday scan), prices Alpaca-only per the charter. **Scheduled live snapshot refresh (Phase 3 M2, `routines/live-snapshot-refresh.md`):** a **deterministic** (not `claude -p`) read-only routine — the routine endpoint calls `refreshLiveAccount` (`account.ts`) directly, which reads the Agentic account via the read-only `claude` CLI path (`get_portfolio` / `get_equity_positions`, no order tool), enriches with the Alpaca mark, persists a fresh `account: "live"` snapshot, and runs the drawdown kill. `summarizeLiveRefresh` maps the result to the RunLog status — a fallback-to-saved/no-snapshot read is `error` so the dead-man + phone alert fires (a failed/stale live read is surfaced). It runs Mon–Fri 07:55 / 12:25 / 15:55 ET (before pre-market, before midday management, and an afternoon pull) so the routines and dashboard read current holdings instead of a manual Refresh. **No order path, no gate change.** Freshness is surfaced on the LIVE panel via the pure `snapshotFreshness` helper (`src/lib/snapshot-freshness.ts`) — a snapshot older than `SNAPSHOT_STALE_AFTER_MINUTES` (6h, wider than the largest refresh gap) shows a **stale** badge. The `install-routines.sh` job format now supports multiple fire times per routine (comma-separated `HH:MM`).
- **Reliability:** `src/lib/server/notify.ts` provides the dead-man switch (healthchecks.io per-routine slug pings on start/success/fail) and phone heartbeats (ntfy / Pushover) on routine start/finish and blocked orders. Both are **fail-soft** (an alert failure must never crash a routine) and **default off** (no env → no-op). Every routine runs through the one trigger endpoint, so the **live** routines (`live-snapshot-refresh`, `live-position-management`) get a dead-man slug + heartbeats automatically — no separate wiring. `scripts/watchdog.sh` supervises the optional news scout, restarting it on crash with capped backoff.
- **Always-on services (Phase 3 M4, `scripts/install-services.sh`):** an unattended live desk needs the dashboard server up (the routines POST to it) and the backup to run itself. The installer writes two launchd plists (writes only; loading is deliberate, like `install-routines.sh`): **`com.tradingdesk.dashboard`** runs `scripts/start-server.sh` with **KeepAlive** (auto-restart on crash) + **RunAtLoad** (survives login/reboot), so a killed server returns and the scheduled routines + the live refresh keep firing across a crash/reboot — it codifies the supervised dashboard service in the repo (same label as the hand-made one → idempotent, one server); **`com.tradingdesk.backup`** runs `scripts/backup.sh` **daily at 03:10** so the encrypted R2 backup runs unattended. `start-server.sh` runs the production server (`pnpm start`; `SERVER_CMD="pnpm dev"` to supervise dev), loading Node via nvm + `.env` since launchd has a minimal environment. The **live drawdown kill switch** (`enforceLiveDrawdownKill`) is wired into every fresh live snapshot (scheduled refresh or manual), latching live OFF + alerting past −10%. Documented in `scripts/README.md`; the preflight check reports whether the server + backup services are installed/loaded.

## LLM runtime
- Agents run as **Claude Code (Max plan)** and **Codex CLI (Codex Pro)**, used as harnesses and invoked as subprocesses (`claude -p`, `codex exec`).
  - **`codex` resolution gotcha (learned 2026-06-26).** The red-team gate spawns `CODEX_BIN` (default `codex`, PATH-resolved — `src/lib/server/red-team.ts`). On this Mac an **nvm npm-global `@openai/codex`** is first on the dashboard server's PATH but its vendored binary is **missing** (empty `…/@openai/codex/vendor/<arch>/codex/`), so a bare `codex` spawn dies with `spawn … ENOENT` and the gate **fails closed → "defaulting to NO"** (the verdict is correct; the cause is the broken binary). The working CLI is **Homebrew `codex` (`/opt/homebrew/bin/codex`, 0.141.0)**, which supports `codex exec [PROMPT]`. Fix = pin it: `CODEX_BIN=/opt/homebrew/bin/codex` in **`.env.local`** (gitignored; loaded above `.env`). Alternatives: `npm uninstall -g @openai/codex` so a bare `codex` resolves to Homebrew, or repair the npm install's vendored binary. `.env.local` is machine-local, so set this per machine.
- **Do not** wire the app to metered LLM APIs for the agent runtime. The subscriptions are the runtime — that is a deliberate cost decision; keep it.
- **One sanctioned metered-API exception:** the optional **Perplexity `finance_search`** research provider (fundamentals / earnings / analyst / catalyst data). Must use the **Agent API** (`POST /v1/agent` with `tools:[{type:"finance_search"}]`) — *not* the Sonar chat-completions endpoint. It is **default-off** (`RESEARCH_PROVIDER=off`), sits behind a swappable `ResearchProvider` interface, is **hard-capped per day** (`PERPLEXITY_DAILY_CALL_CAP`), and is used for **research only — never order pricing or execution** (Alpaca stays the source of truth for prices). This is Perplexity's pay-as-you-go Agent API, *not* the Pro app subscription. Key in `.env` (`PERPLEXITY_API_KEY`).
- **Real-time price data = Alpaca only.** Do not add Yahoo Finance (no official API; 15–20 min delays; no streaming; fragile/ToS-limited). If more real-time depth is needed, upgrade Alpaca's market-data plan or use a proper provider (e.g. Polygon). Implemented in `src/lib/server/research/`: the daily cap is enforced **in code** (a persisted per-day counter in `data/research/`) before any request; entries that used it are tagged `research:perplexity` for later evaluation. Three callers share that one in-code cap: the pre-market routine (via `/api/research/finance`), the **user-initiated** symbol-view highlights button (via `/api/symbol/[ticker]/highlights` — one symbol per click, default-off, falls back to the link-outs when off/capped), and the **user-initiated** "analyze a symbol" pipeline (Phase 3 M2, `src/lib/server/analyze-symbol.ts` via `getSymbolResearch`, one symbol per click; prices stay Alpaca-only, research is best-effort + capped, falls back to the technicals alone when off/capped). Nothing calls it automatically per-ticker. **Analyze-a-symbol (`POST /api/proposals/analyze`)** runs the full proposal pipeline for a human-entered ticker — Alpaca technicals + capped research → a deterministic technically-anchored proposal builder (`src/lib/proposal-builder.ts`, injectable so the analyst step can later be an LLM) → the **risk rails + the red-team** → persisted as a review candidate tagged `manual-request`. It **places nothing**: a paper request is a normal paper proposal, a live request is approvable through the same gated approval path (gate closed → dry-run sink), and a weak pick is flagged by the gates, never rubber-stamped.

## Brokers
- **Live-first (2026-06-25 reorientation).** The **live Robinhood Agentic account is the desk's focus**: the dashboard defaults to the live view, discovery proposes for the live account, and the docs lead with live. The Alpaca **paper** desk is **secondary plumbing** — the gate-closed **dry-run sink** an approved order routes to until the gates open, plus a proving ground / data source. Don't reframe paper as the focus. Safety is unchanged: the app never auto-trades, every live order is human-approved, and the two gates stay human-only. See `planning/live-first-reorientation-spec.md` + `strategy/charter.md`.
- **Alpaca** = the dry-run sink + market-data source + (secondary) paper proving ground. **Real-time price data = Alpaca only** (charter).
- **Robinhood Agentic** = the live account (human-approved execution), via its MCP (`https://agent.robinhood.com/mcp/trading`).
- **Fidelity** = excluded; no retail automation API.
- **Two-gate live safety:** (1) the Robinhood Agentic account allows agent trading; (2) a one-time human `settings.json` allow-list edit enables order tools. An agent may **never** grant itself order permission. Per-trade approval stays ON until paper results justify otherwise.
- **Robinhood read-only client (Phase 3 M1, `src/lib/server/robinhood.ts`):** wires the dashboard LIVE panel to the real account, read-only — its `READ_ONLY_TOOLS` allow-list (`get_portfolio`, `get_equity_positions`) has no order tool by construction. **Transport = the host `claude` CLI, not a token.** The Agentic MCP authenticates via OAuth (the `claude mcp add` + browser authorize flow), so there is **no static bearer token** — the dashboard reads the account by spawning `claude -p` (argv, shell-free, same host-CLI pattern as chat/red-team) with the read-only tools allow-listed and **every order/enumeration tool `--disallowedTools`** (`buildPortfolioCliCommand`, unit-tested). **Privacy (Agentic-only):** it reads only the ONE account named by `ROBINHOOD_AGENTIC_ACCOUNT_NUMBER` and never calls `get_accounts` (in `FORBIDDEN_TOOLS`), so other linked accounts are never fetched or displayed. **Default-off:** with no `ROBINHOOD_AGENTIC_ACCOUNT_NUMBER` the resolver (`getLiveAccount` in `account.ts`) returns `disconnected` and the panel shows **LIVE TRADING: OFF**. When connected it persists each live snapshot to `data/snapshots/` (`account: "live"`) and falls back to the last persisted snapshot if the CLI read times out/fails (the `claude` spawn is slow, ~tens of seconds — the read is best-effort, never blocks the page). The fetch is injectable so mapping/resolver are tested without spawning a CLI or touching a live account. The Positions page renders a labelled LIVE section; the overview LIVE panel lists each live position.
- **Read-only order history for coaching (Phase 3 M2):** the read-only client's `READ_ONLY_TOOLS` allow-list also includes **`get_equity_orders`** — read-only order *history*, which places nothing. It exists so the desk can ingest the human's **manual live trades** for coaching: `getRobinhoodLiveTrades`/`buildOrdersCliCommand` (`robinhood.ts`) read filled orders for the ONE Agentic account (order-placement + `get_accounts` stay in `FORBIDDEN_TOOLS`, asserted by `robinhood.test.ts`), and `syncLiveTrades` (`live-trades.ts`) journals each new fill as a `account: "live"`, `manual: true` decision-journal entry — idempotent via `data/control/live-trades.json`, default-off + best-effort like `getLiveAccount`, triggered by the Coaching page's "Sync live trades" button (`/api/live/sync-trades`). This is **read-only**: it never places an order and the harness order gate stays closed. Coaching reviews these live trades behavior-only (`account: "live"` coaching entries); the paper-desk evaluation stays uncontaminated.
- **Robinhood market-data fundamentals (symbol page, free):** a separate `MARKET_DATA_TOOLS` allow-list (`get_equity_fundamentals` only) feeds the symbol page's stats grid + company profile from Robinhood **for free** (no metered Perplexity call). Unlike the account reads it is **symbol-scoped, not account-scoped** — `buildFundamentalsCliCommand`/`getRobinhoodFundamentals` reference **no account number**, the prompt forbids reading any account, and `get_accounts` + every order tool stay in `FORBIDDEN_TOOLS` (asserted by `robinhood.test.ts`). It supplies market cap, P/E, dividend yield, CEO, employees, sector, industry, and the company blurb; it does **not** supply EPS, exchange, IPO date, or analyst consensus — those fall back to Perplexity. A second `MARKET_DATA_TOOLS` entry, **`get_index_quotes`**, supplies the live **VIX** level for the emergency-stop rail (the free Alpaca stock feed has no `^VIX`): `buildVixCliCommand`/`getRobinhoodVix` (`robinhood.ts`) reference **no account** (least-privilege: each command allow-lists only its own tool, `get_accounts` + order tools forbidden, asserted by `robinhood.test.ts`), and `market-conditions.ts` caches the value for ~10 min (`data/control/market-conditions.json`) so the slow spawn doesn't block every approval, fail-soft to a neutral reading. Index level, **not** order pricing — Alpaca stays the price-of-record. **Prices stay Alpaca-only** (charter) — the symbol page never uses Robinhood's OHLC/quote. Best-effort + gated on `hasRobinhoodConnection()`; the spawn is slow (~tens of seconds), so results are cached per-symbol-per-day (see below). Orchestrated by `getSymbolResearch` (`symbol-research.ts`), which prefers Robinhood and uses Perplexity as the metered auto-fallback; the per-day cache (`data/research/cache/`) means a refresh / navigate-back never re-spends.
- **Robinhood advisory proposals (Phase 3 M2, read-only + advisory):** live-account proposals are **advisory only** — guidance the human executes manually in Robinhood. They carry `account: "live"` + `advisory: true` (`TradeProposalSchema`), are tagged **`live · advisory · execute manually`** in the UI, and have **no approve-to-execute** action — only review/dismiss (`reviewed`/`dismissed` statuses) via `POST /api/proposals/review`, a status-only endpoint that never imports `live-order.ts`. The single order entry point (`POST /api/live/approve` → `submitTradeApproval` → `routeApprovedOrder`) **refuses** any advisory proposal (`isAdvisoryProposal`, `src/lib/proposal-advisory.ts`) before a broker/sink is touched, so no execution path is reachable from a live-advisory proposal — unit-tested. The **order gate, not the scorecard, is the real-money boundary** — `planning/phase-2-evaluation-scorecard.md` is **advisory only** and never blocks going live or any trade (see "Configurable rails & human override" below). Emit advisory proposals with `recordAdvisoryProposal` (writers.ts), which force-stamps `account: "live"`, `advisory: true`, `status: "pending"`.
- **Human-approved live execution (Phase 3 M5a):** "advisory" is now defined by the **`advisory` flag only**, not by account (`isAdvisoryProposal` = `advisory === true`; the long-standing "any live proposal is advisory" fail-safe is **retired** — the *order gate* is the real-money boundary, not the flag). A live proposal can now be **approvable** (`advisory: false`, emitted by `recordApprovableLiveProposal`): the human approves it and the app routes it through `submitTradeApproval` → `routeApprovedOrder`. **Gate CLOSED (shipped) → the dry-run sink (paper/mock), never Robinhood; gate OPEN → real Robinhood.** Per-trade human approval is always required and the app never auto-trades; `assertLiveOrderAllowed` still fails closed, so opening it stays a deliberate human two-gate action. The approval risk re-check is account-aware (a live order is sized against the live snapshot). Manual guidance (`recordAdvisoryProposal`, `advisory: true`) is unchanged: review/dismiss only, no order path. **M5b — order placement.** The real Robinhood `place_equity_order` is now implemented in `live-order.ts` (`buildPlaceOrderCliCommand` + `defaultRobinhoodPlaceOrder`): same host-`claude`-CLI transport as the read-only client, allow-listing ONLY `place_equity_order` (account-scoped) and disallowing every enumeration/cancel/option tool. It **self-gates** (`assertLiveOrderAllowed`) on top of `placeLiveOrder`'s gate, so it stays unreachable until both gates open; injectable + unit-tested (no real order in tests). The live request/response wire shape is verified during a supervised, human-present gate-open test order. **MCP server name is single-sourced** as `ROBINHOOD_MCP_SERVER` (`"robinhood-trading"`) in `gate.ts` — the prior `mcp__robinhood__*` ids in the gate, the revoke script, and the committed `.claude/settings.json` deny-list were wrong; code + scripts are corrected, and **the human must correct the `.claude/settings.json` deny-list ids** (the agent can't edit `.claude/**`). See `planning/two-gate-live-trading.md`.
- **Kill switch & runbook (Phase 3 M6, `scripts/kill-switch.sh`):** one human-run command disables all trading: latches the trading HALT (`data/control/live-halt.json` — written directly so it works server-down), revokes the harness order permission via `scripts/revoke-order-permission.mjs` (order tools out of `allow`, forced into `deny`; idempotent — the human runs it since the agent can't edit `.claude/**`), and unloads the `com.tradingdesk.*` launchd jobs + stops the news scout. Enforced in code too: `isTradingHalted()` makes the routine endpoint (`/api/routines/[id]`) log a `skipped` run and return `503` while halted. Incident steps live in `planning/incident-runbook.md`.
- **Live caps & breakers (Phase 3 M4, `src/lib/server/live-guards.ts`):** live-only guardrails on the funded account, layered on the Phase 2 risk engine. Policy caps live in the charter (`LIVE_LIMITS` in `charter.config.ts`, mirrored in `charter.md`, tripwired by `charter-config.test.ts`): **$100/week funding cap**, **$500 account exposure ceiling**, **−10% drawdown kill**. `evaluateLiveCaps` blocks an over-exposed / over-funded buy on the **live** order path (journaled `rules` rejection → `blocked-caps`); it does not constrain the paper dry-run sink. `enforceLiveDrawdownKill` (wired into `getLiveAccount`) latches live OFF via `disconnectLive` and fires a phone/dead-man alert when a fresh live snapshot breaches the threshold — idempotent + fail-soft. The weekly funding tracker (`recordDeposit`/`checkFundingDeposit`, `data/control/funding.json`) bounds the human's own deposits. Funded capital comes from the live snapshot or `LIVE_FUNDED_CAPITAL_USD`, never from a charter constant.
- **Per-trade approval + dry-run sink (Phase 3 M3, `src/lib/server/live-order.ts`):** every live-intent order needs an explicit human approval (full-preview `AlertDialog` on Proposals → `POST /api/live/approve`) recorded **before** any broker call. `submitTradeApproval` re-runs the risk gate, blocks a red-team `reject`, then `routeApprovedOrder` sends it by gate state: **closed → the dry-run sink** (Alpaca paper if keyed, else an in-process mock) — never Robinhood; **open (M5) → the live Robinhood path** (wired but unreachable while `assertLiveOrderAllowed` fails closed). Approvals/denials are journaled (trade entry vs `human`/`rules`/`codex-redteam` rejection). A sink broker rejection returns a clean `error` outcome (502) and leaves the proposal pending — the order path never 500s. Robinhood order placement lives here, NOT in the read-only `robinhood.ts`.
  - **Order idempotency (pre-live hardening M2, `order-idempotency.ts`).** A double-tap on Approve or a retry after a slow broker call must place **at most once**. `submitTradeApproval` derives a **stable client order id** per approval — the proposal id (the approve route passes `idempotencyKey: proposalId`), never the per-request timestamp — and guards on it **before routing** so the dedupe covers paper, mock, and live. Two layers: an **in-process single-flight** registry (concurrent taps share one in-flight placement — safe because the dashboard is a single localhost process) and a **persisted placement record** (`data/control/placed-orders/<hash>.json`) so a later retry (even after a restart) returns the original `{ destination, brokerOrderId, journalId }` with `idempotent: true` instead of re-placing. The client order id is also passed to Alpaca paper as `client_order_id` (broker-level dedup). Only **placed** orders are recorded — a blocked/denied order is not, so re-submitting (e.g. after adding an override) re-evaluates. Unit + integration tested (same approval twice places once on the dry-run and live paths; concurrent double-submit is safe).
  - **Fresh entry levels + staleness guard (fresh-entry-levels M1).** Proposals
    are anchored to the **current Alpaca quote** at analysis (`buildManualProposalDraft`
    takes a `quote`; `getLatestPrice` in `alpaca.ts` resolves it snapshot→last-bar),
    so a stale daily-bar close can't produce a wrong stop / R:R / size. The user's
    **"Refresh levels"** action (`POST /api/proposals/[id]/refresh-levels` →
    `refreshProposalLevels`) re-anchors every lens off a fresh quote and re-runs
    each red-team, overwriting the same proposal in place (no new metered research).
    `evaluateApprovalBlocks` adds a **staleness block** (`ApprovalBlocks.staleLevels`):
    it reads the current quote (`quoteOf` seam, fail-soft to null) and, when the
    entry has drifted beyond `STALE_DRIFT_THRESHOLD` (1.5%, `src/lib/price-freshness.ts`),
    blocks the order. **This is a real-money correctness gate, checked first and
    NOT clearable by an override comment** — the only remedy is a re-anchor
    (`submitTradeApproval` returns `blocked-stale`, journaled as a `rules`
    rejection `rule:stale-levels`). It is distinct from the red-team's qualitative
    stale-entry catch (kept as belt-and-suspenders). Both `quoteOf` and the quote
    fetch are injectable + unit-tested.
  - **Approval-time risk context uses real inputs (pre-live hardening M1).** `evaluateApprovalBlocks` builds its `RiskContext` from real values, not placeholders, so **every** charter rail fires at per-trade approval: **`ordersToday`** comes from the persisted per-ET-day order counter (`order-counter.ts`, `data/control/order-counter.json`) incremented at each placement, so the **daily-order cap** (≤6/day) holds across the paper batch + every approval and across runs in a day; **`spyIntradayChangePct` + `vix`** come from a live Alpaca read (`market-conditions.ts`) at approval so the **emergency stop** (SPY −2% / VIX>30) fires. Both are injectable seams (tests) and **fail-soft to a neutral reading that trips no rail** — a transient market-data hiccup must not silently block every order. **VIX caveat:** the free Alpaca IEX/SIP stock feed has no `^VIX` (returns "asset not found"), so VIX is sourced through an injectable seam and defaults neutral until a VIX feed is wired; the **SPY arm is always live** and is the dominant broad-stress signal. Drawdown/high-water still come from the snapshot.
- **Two-gate enforcement (Phase 3 M2, `src/lib/server/gate.ts`):** `liveEnabled = brokerGateOpen && harnessGateOpen && !disconnected`. Broker gate = `ROBINHOOD_BROKER_TRADING_ENABLED=1` (human attestation of the Robinhood account setting); harness gate = the order tools allow-listed (and not denied) in `.claude/settings.json`. `assertLiveOrderAllowed()` is the single fail-closed code gate every live-order path must clear. The agent can self-enable **neither**: the module has no enable export, and `.claude/settings.json` denies the agent's `Edit`/`Write` on `.claude/**` (the broker gate lives in Robinhood, off-repo). `disconnectLive()` is the only state change — halt-only, never arm. Procedure: `planning/two-gate-live-trading.md`. The committed `.claude/settings.json` deny-list is shared; `.claude/settings.local.json` (local allow-list) is gitignored.

- **Configurable rails & human override (Phase 3 M7).** The risk rails are the
  user's own controls on the user's own funded account — they are **configurable
  and per-trade overridable**, not immovable. Defaults stay safe; loosening one
  is always explicit, acknowledged, and logged.
  - **Risk settings page (`/risk`).** The user can adjust or disable each of the
    five configurable rails — per-position size cap, daily order cap, drawdown
    halt, stop-required, universe. Overrides persist in a single JSON state file
    (`data/control/risk-settings.json`, an internal state file like the halt latch
    / funding tracker — *not* a `data/` artifact contract; `RiskSettingsSchema`).
    `src/lib/server/risk-settings.ts` (`readRiskSettings` / `writeRiskSettings`,
    edited via the local data-state `POST /api/risk-settings`) and the pure
    `effectiveRiskConfig` overlay turn the settings into the **effective**
    `{ limits, skipRules }` the gate evaluates: an adjusted number overrides the
    charter value; a disabled rail is added to `skipRules` and dropped by
    `evaluateOrder(order, ctx, limits, { skipRules })`. The charter constants
    remain the **safe defaults** (the file only ever *overrides* them; default =
    charter, so the gate stays hard by default), and "reset to charter defaults"
    is one click. Disabling a rail is explicit (a visible OFF state + warning) and
    recorded (persisted with `updatedAt`). The overlay is applied at the
    **approval gate only** (`evaluateApprovalBlocks`); the autonomous pipeline
    still evaluates against the charter defaults, and the human reviews/overrides
    at approval.
  - **Per-trade override (red-team reject OR rail/cap violation).** A red-team
    `reject` and a rail/live-cap violation no longer hard-block at approval time —
    each is **human-overridable via a 2-step flow** driven by the read-only
    `POST /api/live/approve/precheck` (which runs `evaluateApprovalBlocks` with no
    side effects): (1) the order + the blocking reason (red-team reasoning / which
    rail + message) are shown prominently in the (widened) approve dialog; (2) a
    typed **non-empty justification comment** is required and "Override & approve"
    stays disabled until one is entered. `submitTradeApproval` re-enforces it in
    code via `hasValidOverride` (a blank comment never bypasses a block) — the one
    safeguard the software keeps. Unit-tested both ways (blank comment still
    blocks; a real comment overrides + logs).
  - **Everything logged.** Going live, every real order, and **every override**
    (red-team or rail/cap) is written to the decision journal — the placed-trade
    entry carries `human-override` + `override:<rule>` tags and a
    `HUMAN OVERRIDE — comment: "…". Overrode: …` line in the body, the user's own
    audit trail and model-training signal. No override is silent.
  - **Unmistakable LIVE state.** When both gates are open, an app-wide
    `LiveBanner` (`src/components/live-banner.tsx`, in the root layout, gated on
    `getLiveTradingStatus().liveEnabled`) shows a danger-toned "LIVE TRADING IS
    ON" strip on every page; the approve dialog also reframes to "place a REAL
    order". Rendering the banner changes no gate.
  - **The scorecard is advisory, never blocking.** `planning/phase-2-evaluation-scorecard.md`
    informs the user; it is **not** a precondition for opening the gate or placing
    any trade. The two human gates are the only real-money boundary.
  - **The one invariant kept:** the agent **never self-opens the live gate, never
    funds the account, and never places a real order without the user's explicit
    per-trade approval.** Configurable rails and overrides are all *human* actions
    surfaced in the UI; none of them lets the *software* (a bug, stray click, or
    prompt-injection) arm real-money trading by itself. Opening the gate stays the
    deliberate two-gate human action (env + `.claude/settings.json`).

## Secrets
- `.env` is gitignored. Real values live in macOS Keychain / 1Password. Keep `.env.example` current.
- Never print secrets to logs, journals, or the dashboard.

## Data & backups (three tiers)
1. **Code + docs + strategy** → private git remote (`origin`).
2. **Secrets** → never in git; Keychain / 1Password.
3. **`data/`** (journals, snapshots, chats) → **Cloudflare R2, client-side encrypted via `rclone crypt`**, gitignored. Daily job lives in `scripts/` (`backup.sh` / `restore.sh`; setup in `scripts/README.md`). The scripts build the rclone config from `.env` env vars at runtime — **no secrets in any rclone config file**; crypt password/salt live in `.env` (`R2_CRYPT_PASSWORD` / `R2_CRYPT_SALT`).
- Never live-symlink the working tree into Google Drive (it corrupts `.git`). Back up snapshots, not the live tree.
