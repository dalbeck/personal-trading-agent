# Infrastructure Rules

## Hosting & processes
- Runs **natively on macOS**. No Docker, no DDEV. The dashboard backend must reach the host-authenticated `claude` and `codex` CLIs and the local filesystem directly ‚Äî a container boundary would break that.
- Keep long-running processes alive with **pm2** or **launchd**, not a container.
- The dashboard is a **local** app (localhost). Never expose it publicly ‚Äî it can read all accounts and (eventually) place trades.
- **Scheduled routines:** one launchd job per routine runs `scripts/run-routine.sh <id>`, which POSTs to `/api/routines/<id>` on the always-on dashboard server (the server runtime owns the lockfile, the code-gated execution pipeline, Alpaca paper, and the `codex` red-team). The endpoint runs under a single-instance lockfile and writes a `RunLog` to `data/logs/`. The trigger endpoint places paper orders, so it's gated by `ROUTINE_TRIGGER_TOKEN` (bearer) and must stay localhost-only. `scripts/install-routines.sh` writes the plists but never loads them ‚Äî loading is a deliberate human act. **Execution is code-gated: every order clears the risk rails + red-team before Alpaca; the LLM proposes, it never executes.**
- **Reliability:** `src/lib/server/notify.ts` provides the dead-man switch (healthchecks.io per-routine slug pings on start/success/fail) and phone heartbeats (ntfy / Pushover) on routine start/finish and blocked orders. Both are **fail-soft** (an alert failure must never crash a routine) and **default off** (no env ‚Üí no-op). `scripts/watchdog.sh` supervises the optional news scout, restarting it on crash with capped backoff.

## LLM runtime
- Agents run as **Claude Code (Max plan)** and **Codex CLI (Codex Pro)**, used as harnesses and invoked as subprocesses (`claude -p`, `codex exec`).
- **Do not** wire the app to metered LLM APIs for the agent runtime. The subscriptions are the runtime ‚Äî that is a deliberate cost decision; keep it.
- **One sanctioned metered-API exception:** the optional **Perplexity `finance_search`** research provider (fundamentals / earnings / analyst / catalyst data). Must use the **Agent API** (`POST /v1/agent` with `tools:[{type:"finance_search"}]`) ‚Äî *not* the Sonar chat-completions endpoint. It is **default-off** (`RESEARCH_PROVIDER=off`), sits behind a swappable `ResearchProvider` interface, is **hard-capped per day** (`PERPLEXITY_DAILY_CALL_CAP`), and is used for **research only ‚Äî never order pricing or execution** (Alpaca stays the source of truth for prices). This is Perplexity's pay-as-you-go Agent API, *not* the Pro app subscription. Key in `.env` (`PERPLEXITY_API_KEY`).
- **Real-time price data = Alpaca only.** Do not add Yahoo Finance (no official API; 15‚Äì20 min delays; no streaming; fragile/ToS-limited). If more real-time depth is needed, upgrade Alpaca's market-data plan or use a proper provider (e.g. Polygon). Implemented in `src/lib/server/research/`: the daily cap is enforced **in code** (a persisted per-day counter in `data/research/`) before any request; entries that used it are tagged `research:perplexity` for later evaluation. Two callers share that one in-code cap: the pre-market routine (via `/api/research/finance`) and the **user-initiated** symbol-view highlights button (via `/api/symbol/[ticker]/highlights` ‚Äî one symbol per click, default-off, falls back to the link-outs when off/capped). Nothing calls it automatically per-ticker.

## Brokers
- **Alpaca** = paper trading and the default build/proving ground for all development.
- **Robinhood Agentic** = optional live execution (Phase 3 only), via its MCP (`https://agent.robinhood.com/mcp/trading`).
- **Fidelity** = excluded; no retail automation API.
- **Two-gate live safety:** (1) the Robinhood Agentic account allows agent trading; (2) a one-time human `settings.json` allow-list edit enables order tools. An agent may **never** grant itself order permission. Per-trade approval stays ON until paper results justify otherwise.
- **Robinhood read-only client (Phase 3 M1, `src/lib/server/robinhood.ts`):** wires the dashboard LIVE panel to the real account via `get_portfolio` only ‚Äî its `READ_ONLY_TOOLS` allow-list has no order tool by construction. **Privacy (Agentic-only):** the MCP can read every linked account, but the desk reads only `get_portfolio` (the single Agentic account) ‚Äî never account-enumeration tools (`FORBIDDEN_TOOLS`, unit-tested). **Default-off:** with no `ROBINHOOD_MCP_TOKEN` the resolver (`getLiveAccount` in `account.ts`) returns `disconnected` and the panel shows **LIVE TRADING: OFF**. When connected it persists each live snapshot to `data/snapshots/` (`account: "live"`) so the panel and the agent share one source of truth. The MCP round-trip is injectable so mapping/resolver are tested without a network or a live account. The Positions page renders a labelled LIVE section; the overview LIVE panel lists each live position.
- **Robinhood advisory proposals (Phase 3 M2, read-only + advisory):** live-account proposals are **advisory only** ‚Äî guidance the human executes manually in Robinhood. They carry `account: "live"` + `advisory: true` (`TradeProposalSchema`), are tagged **`live ¬∑ advisory ¬∑ execute manually`** in the UI, and have **no approve-to-execute** action ‚Äî only review/dismiss (`reviewed`/`dismissed` statuses) via `POST /api/proposals/review`, a status-only endpoint that never imports `live-order.ts`. The single order entry point (`POST /api/live/approve` ‚Üí `submitTradeApproval` ‚Üí `routeApprovedOrder`) **refuses** any advisory proposal (`isAdvisoryProposal`, `src/lib/proposal-advisory.ts`) before a broker/sink is touched, so no execution path is reachable from a live-advisory proposal ‚Äî unit-tested. The harness order gate stays **closed**; automated execution (M5) remains gated on a passing `planning/phase-2-evaluation-scorecard.md`. Emit advisory proposals with `recordAdvisoryProposal` (writers.ts), which force-stamps `account: "live"`, `advisory: true`, `status: "pending"`.
- **Kill switch & runbook (Phase 3 M6, `scripts/kill-switch.sh`):** one human-run command disables all trading: latches the trading HALT (`data/control/live-halt.json` ‚Äî written directly so it works server-down), revokes the harness order permission via `scripts/revoke-order-permission.mjs` (order tools out of `allow`, forced into `deny`; idempotent ‚Äî the human runs it since the agent can't edit `.claude/**`), and unloads the `com.tradingdesk.*` launchd jobs + stops the news scout. Enforced in code too: `isTradingHalted()` makes the routine endpoint (`/api/routines/[id]`) log a `skipped` run and return `503` while halted. Incident steps live in `planning/incident-runbook.md`.
- **Live caps & breakers (Phase 3 M4, `src/lib/server/live-guards.ts`):** live-only guardrails on the funded account, layered on the Phase 2 risk engine. Policy caps live in the charter (`LIVE_LIMITS` in `charter.config.ts`, mirrored in `charter.md`, tripwired by `charter-config.test.ts`): **$100/week funding cap**, **$500 account exposure ceiling**, **‚àí10% drawdown kill**. `evaluateLiveCaps` blocks an over-exposed / over-funded buy on the **live** order path (journaled `rules` rejection ‚Üí `blocked-caps`); it does not constrain the paper dry-run sink. `enforceLiveDrawdownKill` (wired into `getLiveAccount`) latches live OFF via `disconnectLive` and fires a phone/dead-man alert when a fresh live snapshot breaches the threshold ‚Äî idempotent + fail-soft. The weekly funding tracker (`recordDeposit`/`checkFundingDeposit`, `data/control/funding.json`) bounds the human's own deposits. Funded capital comes from the live snapshot or `LIVE_FUNDED_CAPITAL_USD`, never from a charter constant.
- **Per-trade approval + dry-run sink (Phase 3 M3, `src/lib/server/live-order.ts`):** every live-intent order needs an explicit human approval (full-preview `AlertDialog` on Proposals ‚Üí `POST /api/live/approve`) recorded **before** any broker call. `submitTradeApproval` re-runs the risk gate, blocks a red-team `reject`, then `routeApprovedOrder` sends it by gate state: **closed ‚Üí the dry-run sink** (Alpaca paper if keyed, else an in-process mock) ‚Äî never Robinhood; **open (M5) ‚Üí the live Robinhood path** (wired but unreachable while `assertLiveOrderAllowed` fails closed). Approvals/denials are journaled (trade entry vs `human`/`rules`/`codex-redteam` rejection). A sink broker rejection returns a clean `error` outcome (502) and leaves the proposal pending ‚Äî the order path never 500s. Robinhood order placement lives here, NOT in the read-only `robinhood.ts`.
- **Two-gate enforcement (Phase 3 M2, `src/lib/server/gate.ts`):** `liveEnabled = brokerGateOpen && harnessGateOpen && !disconnected`. Broker gate = `ROBINHOOD_BROKER_TRADING_ENABLED=1` (human attestation of the Robinhood account setting); harness gate = the order tools allow-listed (and not denied) in `.claude/settings.json`. `assertLiveOrderAllowed()` is the single fail-closed code gate every live-order path must clear. The agent can self-enable **neither**: the module has no enable export, and `.claude/settings.json` denies the agent's `Edit`/`Write` on `.claude/**` (the broker gate lives in Robinhood, off-repo). `disconnectLive()` is the only state change ‚Äî halt-only, never arm. Procedure: `planning/two-gate-live-trading.md`. The committed `.claude/settings.json` deny-list is shared; `.claude/settings.local.json` (local allow-list) is gitignored.

## Secrets
- `.env` is gitignored. Real values live in macOS Keychain / 1Password. Keep `.env.example` current.
- Never print secrets to logs, journals, or the dashboard.

## Data & backups (three tiers)
1. **Code + docs + strategy** ‚Üí private git remote (`origin`).
2. **Secrets** ‚Üí never in git; Keychain / 1Password.
3. **`data/`** (journals, snapshots, chats) ‚Üí **Cloudflare R2, client-side encrypted via `rclone crypt`**, gitignored. Daily job lives in `scripts/` (`backup.sh` / `restore.sh`; setup in `scripts/README.md`). The scripts build the rclone config from `.env` env vars at runtime ‚Äî **no secrets in any rclone config file**; crypt password/salt live in `.env` (`R2_CRYPT_PASSWORD` / `R2_CRYPT_SALT`).
- Never live-symlink the working tree into Google Drive (it corrupts `.git`). Back up snapshots, not the live tree.

## Repo layout
`planning/` ¬∑ `src/` (dashboard) ¬∑ `strategy/` (charter, playbook, rules) ¬∑ `routines/` (prompt files + scheduler) ¬∑ `scripts/` (backup, healthcheck, lockfile) ¬∑ `data/` (gitignored, R2-backed)
